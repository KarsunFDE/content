# W05-SA-3 — Bedrock managed-services migration cost-benefit

## Client constraint

> *"AWS managed services just lit up per the D-050 carve-out — Bedrock Knowledge Bases, Agents-for-Bedrock, OpenSearch Managed. The agency CTO wants to know whether to migrate our existing custom RAG (`/rag/clause-search`) to Bedrock Knowledge Bases and our existing LangGraph multi-agent flow (`POST /agent/intake-triage`) to Agents-for-Bedrock. The pitch is FedRAMP boundary cleanliness — AWS-managed = thinner SSP, faster ATO. The anti-pitch is vendor lock-in + losing engineering control of the surface. Walk me through when each migration is worth it and when it's not."*
>
> — Agency CTO via the Karsun engagement lead, W5 Mon 2026-06-22

## The programme default

**Hybrid** — Bedrock Knowledge Bases for new corpora (e.g., agency-supplement clauses going forward); custom Atlas Vector Search retained for FAR/DFARS (cohort owns the surface; debuggability matters when the OIG asks why a clause was cited). LangGraph retained for the high-HITL `intake-triage` flow (HITL #5's interrupt-node behaviour doesn't map 1:1 to Bedrock Agents' user-confirmation API). Agents-for-Bedrock used for lower-HITL future flows.

This is **a position**, not a foregone conclusion. The pair may argue for full migration or no migration with sufficient rationale.

## Candidate migrations the pair researches Wed W5

### Migration 1 — `/rag/clause-search` to Bedrock Knowledge Bases

**Current state.** `ai-orchestrator/app/rag/clause_search.py` does hybrid retrieval (dense + sparse + custom re-ranker) over MongoDB Atlas Vector Search. ~3,500 pages of FAR + DFARS clauses chunked + embedded (per D-060 W2 corpus).

**Managed state.** [Bedrock Knowledge Bases, retrieved 2026-05-23, https://docs.aws.amazon.com/bedrock/latest/userguide/knowledge-base.html] manages chunking + embedding + retrieval + citation surfacing. Cohort uploads corpus to S3; Bedrock handles the rest.

| Dimension | Custom Atlas | Bedrock KB |
|-----------|--------------|------------|
| Chunking control | Full (cohort picked W2) | Limited (Bedrock-default; some override) |
| Citation surface | Custom UX (W2) | Built-in citations API |
| Multi-tenant filter | `$vectorSearch` filter on agency_id | Metadata filter on KB chunk metadata |
| FedRAMP boundary | Atlas not AWS-managed (D-050 carve-out N/A) | AWS-managed; clean GovCloud story |
| Vendor lock-in | MongoDB Atlas | AWS Bedrock |
| Debug when retrieval fails | High (cohort code) | Lower (managed pipeline) |
| Latency | ~100ms (cached embedding) | ~200ms (KB-managed) |
| Cost shape | Atlas tier + Bedrock embed call | Per-query + storage |
| Re-tokenization control | Pair triggers + observes | Bedrock-managed cadence |

**When migration wins.** Pair has limited engineering bandwidth for vector-search ops; the agency is hard-wired to AWS for the SSP; the corpus is *new* and pair hasn't already invested chunking work; debuggability is less critical (e.g., the citation surface is the OIG-visible artifact, not the retrieval internals).

**When migration loses.** Pair already invested chunking + re-ranker work that beats Bedrock-default quality on the FAR/DFARS specific corpus; debuggability matters (W5 Wed Pattern 3 root-cause analysis on retrieval requires owning the surface); latency matters (the 100ms→200ms is felt at the CO drafting UX).

### Migration 2 — `POST /agent/intake-triage` to Agents-for-Bedrock

**Current state.** W3 multi-agent flow: proposal intake → triage agent → evaluator-routing agent → CO interrupt-node (HITL #5). LangGraph state machine; cohort-owned interrupts.

**Managed state.** [Bedrock Agents, retrieved 2026-05-23, https://docs.aws.amazon.com/bedrock/latest/userguide/agents.html] manages action groups (tools via OpenAPI schema), session state, user-confirmation API. Cohort exposes solicitation-service + evaluation-service endpoints as action groups.

| Dimension | LangGraph custom | Agents-for-Bedrock |
|-----------|------------------|---------------------|
| HITL interrupt control | Full (interrupt_before / interrupt_after on any node) | Built-in user-confirmation API (action-group-level) |
| Tool authentication | Cohort-owned JWT propagation | IAM-grounded (per action group role) |
| Debug surface | LangGraph state inspection | CloudWatch + Bedrock agent traces |
| Vendor lock-in | LangChain v1.0 | AWS Bedrock (no portability) |
| Re-targetable to non-AWS | Yes (rewrite the graph) | No (Bedrock-only) |
| Cost shape | Per-LLM-call only | Per-LLM-call + agent orchestration markup |
| Match for HITL #7 ADR | Direct (cohort code wires the chosen authority) | Constrained by Bedrock's user-confirmation surface |

**When migration wins.** Flow's HITL pattern aligns with Bedrock's user-confirmation API (low-frequency, action-group-aligned interrupts); IAM-grounded tool auth is preferable to cohort's JWT propagation; portability isn't a concern; pair wants to delegate session state.

**When migration loses.** HITL pattern is per-node (LangGraph's interrupt nodes don't map to action-group-only confirmations); pair already built debug tooling for LangGraph state; cohort needs portability (e.g., Karsun engagement might land on Azure GovCloud where Bedrock isn't an option).

## Pair's deliverable (consolidated worth-it ADR by EOD Wed, executed Thu)

Per `templates/scenario-alternatives.md`, **one ADR per migration** (so 2 ADRs minimum + optional stretch ADR for OpenSearch):

- **Decision** — full / hybrid / no-migrate per migration target.
- **Context** — what the existing surface does well + where it's brittle.
- **Alternatives** — both alternatives (managed full vs no-migrate) evaluated.
- **Consequences** — what could go wrong with each path.
- **Rollback story** — if Cohort #2 needs to reverse, what's the cost?
- **Measurements** — Thu afternoon's migration work includes latency / cost / debuggability measurements. ADRs without numbers are weak.

## Anti-patterns to call out in defense

- **Migrating because it's new.** Bedrock managed services are slick demos. The ADR commits to the migration only if the *measured* delta justifies it.
- **No-migrating because of "we already built it."** Sunk-cost fallacy. If the managed service genuinely wins on every dimension that matters, the cohort migrates.
- **Skipping `agency_id` filter on Bedrock KB.** Multi-tenant boundary is non-negotiable; Item 10 must NOT regress.
- **Treating vendor lock-in as binary.** Lock-in is a spectrum. LangChain v1.0 is also lock-in (different vendor). The question is *which* lock-in fits the engagement's portability constraints.
- **Skipping HITL #7 alignment check on Migration 2.** Bedrock Agents' user-confirmation API is action-group-scoped; cohort's HITL #7 ADR may not map cleanly. The ADR must call out the alignment status.

## Probe questions for Live Defense

1. *"You migrated `/rag/clause-search` to Bedrock KB. Demonstrate that `agency_id` filtering still works — show me a query for one tenant that returns zero results from another tenant's documents."*
2. *"Your worth-it ADR claims 200ms latency on Bedrock KB vs 100ms on custom Atlas. Where did the 100ms come from? Show me the `time` output."*
3. *"You chose to keep LangGraph instead of migrating to Agents-for-Bedrock. What's your contingency if Karsun's next engagement requires a non-AWS deployment? You've still got vendor lock-in to LangChain v1.0."*
4. *"Walk me through how HITL #7's chosen authority (your Wed ADR's option A/B/C) gets enforced in the migrated Bedrock Agents flow vs the retained LangGraph flow. Does the migration weaken HITL #7?"*
5. *"The agency CTO asks: 'if we pick Bedrock KB, what's our story when the OIG asks why a specific FAR clause was retrieved?'. Walk me through the audit trail in both paths."*
