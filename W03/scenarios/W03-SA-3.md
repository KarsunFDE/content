# W03-SA-3 — KG/CG representation: Neo4j vs Postgres recursive CTE vs NetworkX in-process

## 1. The client constraint

> "The CO told me yesterday she wants to ask the platform: 'show me every contract this vendor has won with red CPARs.' That's a graph traversal — Vendor → Award → Contract → Cpar with a rating filter. I'm fine with that. But here's the thing: we already have Postgres + MongoDB Atlas in our stack. Adding a third datastore (Neo4j) means another FedRAMP control surface, another backup story, another on-call rotation. Defend whether we need it. And — if we don't — defend whether Postgres recursive CTEs are strong enough, or whether NetworkX in-process would actually work for our data volumes."
>
> — Karsun delivery lead, captured Wed afternoon W3 scenario-research slot

## 2. The default programme answer

**Postgres recursive CTE** over the existing acquire-gov Postgres. Justification: the data volumes (~100 vendors / ~500 proposals / ~80 active contracts per tenant per `training-project/feature-inventory-target.md`) are *small for a graph*; Postgres recursive `WITH RECURSIVE` syntax handles 4-6 hop traversals well; staying within the existing datastore eliminates a FedRAMP control surface; ops budget unchanged. Trade-off: query authoring is harder (recursive CTE syntax is unfamiliar to many engineers); cohort needs to be trained in the pattern.

## 3. Candidate alternatives the pair evaluates

| Alternative | One-line capability | Maturity |
|-------------|---------------------|----------|
| Neo4j | Purpose-built graph DB; Cypher query language is purpose-fit for traversal; native variable-length path matching | Stable (Community 5.x + Aura cloud); FedRAMP status varies by deployment (verify via /web-research at research time); separate datastore + separate ops |
| NetworkX in-process | Load the graph into Python on every query; run traversal in-process via NetworkX algorithms | Stable Python library; load latency is the variable — scales poorly past O(10k nodes); fine for cohort-scale, breaks at multi-tenant production |

## 4. Comparison dimensions the pair scores (1–5)

For each alternative (and the default Postgres recursive CTE), score on:

- **Capability fit** — can the alternative answer the "every contract this vendor won with red CPARs" query in <200ms p95 at the stated data volumes? Variable-length path queries (3-6 hops) — does the alternative support them naturally?
- **Cost shape** — Neo4j: separate license + separate ops (Aura cloud ~$X/month, self-hosted = ops time). Postgres recursive CTE: zero added cost. NetworkX: zero added infra cost + Python process memory overhead. Project across a 12-month engagement.
- **Operational burden** — backup story, monitoring, failure modes, on-call rotation impact, FedRAMP control delta. The "third datastore" tax.
- **Security posture** — multi-tenant boundary (Item 10) implementation surface. For each: where do you enforce `agency_id` filtering? Audit-log integration story (Item 2).
- **Maturity** — Neo4j: very stable, frequent releases (verify version via /web-research). Postgres recursive CTE: timeless (works in Postgres 8.4+ from 2009). NetworkX: stable Python library.
- **Exit cost** — switching graph backend in 24 months. Neo4j → Postgres recursive CTE = porting Cypher → recursive SQL (significant). Postgres → Neo4j = porting recursive SQL → Cypher (significant). NetworkX → anything = trivial (was never persistent).

## 8. Anti-patterns to call out in defense

- **Citing Neo4j performance claims without a workload anchor.** Neo4j benchmarks are typically against social-graph workloads, not federal-procurement schemas. Apples-to-oranges.
- **Treating NetworkX as "production-viable" at the cohort's data volumes.** It might be — at *one tenant*. Multi-tenant (think 50 agencies) breaks it. The defender should know where the cliff is.
- **Postgres recursive CTE "looks easy" reasoning.** Authoring a 6-hop traversal in recursive SQL is harder than authoring it in Cypher. The defender should have *written one* before defending the call.
- **Ignoring the FedRAMP control surface.** Adding a third datastore adds 6-12 FedRAMP controls. The defender should know which ones.
- **Forgetting agency_id filtering.** In Postgres the filter is `WHERE agency_id = $1` everywhere. In Neo4j it's a `:Vendor {agency_id: $agency_id}` constraint on every match. In NetworkX it's a pre-filter on graph load. Each has a different "where do I forget it" surface.
- **Citing Aura FedRAMP status from memory.** This changes; verify via /web-research at research time.
