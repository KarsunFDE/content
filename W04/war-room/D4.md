## 1. What we're tackling today (and why)

The agency CIO has asked for a Spring Boot 4.x posture statement by Friday's exec brief. The ask isn't "are you done modernizing" — it's "where are you on the path, and what's the 90-day plan." Spring Boot 2.7.18 has been OSS-unsupported since 30 Jun 2023 per `research/spring-boot-2-7-to-4-x-20260525.md` — this is **security work, not aesthetic**.

**Why 4.0 and not 3.5.** The earlier curriculum target was SB 3.5. We retargeted: SB 3.5 OSS support ends 30 Jun 2026 — within days of Cohort #1's W4 finish. Landing the cohort on a version that flips to commercial-only at W5 is a federal-compliance posture failure (federal modernization expects OSS-supported runtimes; commercial-only is a procurement red flag). SB 4.0 OSS-supported through 31 Dec 2026 (~6 months runway). Java 21 (not 17) because Spring Framework 7's recommended runtime is Java 21+ (virtual threads, sealed classes, pattern matching) and Java 21 is the LTS sweet spot. The bigger-hop tooling risk is manageable: OpenRewrite's `UpgradeSpringBoot_4_0` recipe composes the 2.7→3.5 + SF6→SF7 + SS6→SS7 sub-recipes internally, so the path through 3.x happens inside one recipe invocation.

Each pair owns one service in parallel: pair-1 = `solicitation-service`, pair-2 = `evaluation-service`, pair-3 = cross-cutting `javax.*` → `jakarta.*` work in shared modules. Future-hop ADRs (Java 25, SF7 minor releases) clear the D-054 Pass 3 evidence floor (≥1 OpenRewrite dryRun per future hop, ≥1 representative-module compile attempt, named breaking changes, ≥1 hand-edit identified) — pure paper gets cut at Live Defense Friday. EOD: green CI on `stage-4.0` for at least one service.
