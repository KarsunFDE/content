# W04-SA-3 — Modernization sequencing

## 1. The client constraint

> *"For the Friday exec brief, the agency CIO is going to ask why we sequenced the modernization the way we did. 'J21 first then SB 4.0' is the path I'd recommend, but the CIO has read a McKinsey deck that recommends single-stage rip-and-replace. I need the rationale documented — short ADR, two pages, codex-defensible. The 2-stage shape comes from D-054 + the codex Pass debate; if the cohort's modernization shape comes out differently in execution, we need an honest update."*
>
> — Karsun delivery lead, framed Mon W4 Plan Day

## 2. The programme default

**J21 first → SB 4.0 second** (per D-054 5-stage checkpoint shape: `legacy-baseline → stage-J21 → stage-3.0-rewrite → stage-4.0 → final-expected-PR` — the `stage-3.0-rewrite` mid-checkpoint is preserved because the composed `UpgradeSpringBoot_4_0` recipe walks through SB 3.x internally and the cohort needs an inspectable mid-state). Java 21 first because **SB 4.x recommends Java 21 baseline** (Java 17 is the floor; SF7's recommended runtime is Java 21+) — running J21 alone first isolates JDK-introduced failures from Spring-framework-introduced failures. Codex Pass 1 verified this sequencing on the earlier J17/SB 3.5 scope; codex Pass 2 re-affirmed against alternative sequencings. The retarget to J21/SB 4.0 preserves the sequencing logic (the underlying claim — JDK-baseline-required-by-framework-target — is unchanged).

## 3. Candidate alternatives the pair must evaluate

| Alternative | One-line capability | Maturity (per `tech-research` recency policy) |
|-------------|---------------------|-----------------------------------------------|
| **SB 4.0 first → J21 second** | Spring framework upgrade first, JDK upgrade second | **Not viable in practice**: SB 4.x requires Java 17 floor + Java 21 recommended — you can't upgrade SB 4.0 on Java 11. Strawman alternative; the pair must defend why it's a strawman. |
| **Combined single-stage hop** | Run OpenRewrite once with J21 + SB 4.0 (composed `UpgradeSpringBoot_4_0` walking SB 3.x internally) + jakarta migration all at once; one CI cycle | Stable as a pattern but failure-isolation suffers — if `stage-combined` fails, you don't know which dimension broke (JDK vs SB 3.x intermediate vs SF7 vs SS7 vs jakarta namespace). McKinsey "rip-and-replace" framing fits here. |
| *(optional)* **3-stage shape**: `stage-J21 → stage-jakarta-only (no SB) → stage-4.0` | Splits jakarta namespace move from SB upgrade for tighter isolation | More defensive but more stages = more rescue branches = more cohort time. Pair must justify the marginal isolation benefit vs 5-day budget. |

## 4. Comparison dimensions the pair must score

Per `templates/scenario-alternatives.md` §4. **Specific dimensions to weight heavily for this scenario:**

- **Failure isolation** — when (not if) something breaks, how quickly does the pair identify which dimension introduced the failure?
- **Time to green CI** — 5-day budget; multi-stage costs CI cycles.
- **D-054 Pass 3 evidence floor** — does the sequencing support cohort production of `dryRun` evidence per future hop (Java 25, SF7 minor releases)?
- **Rollback cost** — if `stage-4.0` blows up at hour 36, how far back does the rollback go? The intermediate `stage-3.0-rewrite` checkpoint is the natural mid-rollback target since the composed recipe walks through SB 3.x internally.

## 8. Anti-patterns to call out in defence

- **Defending the default without acknowledging the McKinsey framing** — the CIO has read it; the defense must address it head-on, not pretend it doesn't exist.
- **Citing "best practices" without naming the specific SB 4.0 → Java 17/21 baseline requirement** — defense rests on the *technical* constraint (you can't run SB 4.0 on J11; SF7 recommends Java 21+), not on generic sequencing wisdom.
- **Treating the 3-stage shape as "more defensive is always better"** — every additional stage costs a CI cycle + rescue branch + cohort time. Defending a 5-stage shape requires defending the time cost.
- **Not naming the actual Thursday execution outcome** — if the cohort diverged from J21-first → SB-4-second (e.g., one pair did combined-single-stage to save time), the ADR must honestly update.
