# W04-SA-3 — Modernization sequencing

## 1. The client constraint

> *"For the Friday exec brief, the agency CIO is going to ask why we sequenced the modernization the way we did. 'J17 first then SB 3.0' is the path I'd recommend, but the CIO has read a McKinsey deck that recommends single-stage rip-and-replace. I need the rationale documented — short ADR, two pages, codex-defensible. The 2-stage shape comes from D-054 + the codex Pass debate; if the cohort's modernization shape comes out differently in execution, we need an honest update."*
>
> — Karsun delivery lead, framed Mon W4 Plan Day

## 2. The programme default

**J17 first → SB 3.0 second** (per D-054 5-stage checkpoint shape: `legacy-baseline → stage-J17 → stage-3.0-rewrite → stage-3.0 → final-expected-PR`). Java 17 first because **SB 3.0 requires Java 17 baseline** — running J17 alone first isolates JDK-introduced failures from Spring-framework-introduced failures. Codex Pass 1 verified this sequencing; codex Pass 2 re-affirmed against alternative sequencings.

## 3. Candidate alternatives the pair must evaluate

| Alternative | One-line capability | Maturity (per `tech-research` recency policy) |
|-------------|---------------------|-----------------------------------------------|
| **SB 3.0 first → J17 second** | Spring framework upgrade first, JDK upgrade second | **Not viable in practice**: SB 3.0 requires Java 17 — you can't upgrade SB 3.0 on Java 11. Strawman alternative; the pair must defend why it's a strawman. |
| **Combined single-stage hop** | Run OpenRewrite once with J17 + SB 3.0 + jakarta migration all at once; one CI cycle | Stable as a pattern but failure-isolation suffers — if `stage-combined` fails, you don't know which dimension broke. McKinsey "rip-and-replace" framing fits here. |
| *(optional)* **3-stage shape**: `stage-J17 → stage-jakarta-only (no SB) → stage-3.0` | Splits jakarta namespace move from SB upgrade for tighter isolation | More defensive but more stages = more rescue branches = more cohort time. Pair must justify the marginal isolation benefit vs 5-day budget. |

## 4. Comparison dimensions the pair must score

Per `templates/scenario-alternatives.md` §4. **Specific dimensions to weight heavily for this scenario:**

- **Failure isolation** — when (not if) something breaks, how quickly does the pair identify which dimension introduced the failure?
- **Time to green CI** — 5-day budget; multi-stage costs CI cycles.
- **D-054 Pass 3 evidence floor** — does the sequencing support cohort production of `dryRun` evidence per future hop (SB 3.5, SB 4.0)?
- **Rollback cost** — if `stage-3.0` blows up at hour 36, how far back does the rollback go?

## 5. Required pair deliverable (in their pair Project repo)

A short ADR (≤2 pages) at `adrs/W04-SA-3.md`:

- Decision: which sequencing the pair recommends + whether the pair actually executed that sequencing Thu (honest update if it diverged).
- Context: 1-paragraph re-statement of the constraint (agency CIO + McKinsey deck framing).
- Considered: J17-first + SB3-first (strawman) + combined-single-stage + optional 3-stage, scored 1–5.
- Consequences: failure-isolation tradeoffs, rollback cost, time-to-green.
- References: dated citations for every claim, including D-054 + codex Pass 1/2 rationale.

## 8. Anti-patterns to call out in defence

- **Defending the default without acknowledging the McKinsey framing** — the CIO has read it; the defense must address it head-on, not pretend it doesn't exist.
- **Citing "best practices" without naming the specific SB 3.0 → Java 17 baseline requirement** — defense rests on the *technical* constraint (you can't run SB 3.0 on J11), not on generic sequencing wisdom.
- **Treating the 3-stage shape as "more defensive is always better"** — every additional stage costs a CI cycle + rescue branch + cohort time. Defending a 5-stage shape requires defending the time cost.
- **Not naming the actual Thursday execution outcome** — if the cohort diverged from J17-first → SB-3-second (e.g., one pair did combined-single-stage to save time), the ADR must honestly update.
