# W04-SA-1 — Circuit breaker library for Item 3

## 1. The client constraint

> *"After Friday's Mid-Sprint Surprise — which I hope you ship cleanly — we're going to need a circuit-breaker library committed to the W5 AIOps plan. The agency CIO will ask: 'Why this library and not the one your previous prime used?' We need an honest comparison. The constraint isn't 'fastest library' — it's 'library we can defend across a 2-year FedRAMP authorization cycle.' That means: actively maintained, Spring Boot 4.x-compatible (services are on SB 4.0 + Java 21 post-Thu hop), plays nicely with OpenTelemetry, doesn't go EOL or archive on us in 18 months."*
>
> — Karsun delivery lead, framed Wed PM /web-research slot

## 2. The programme default

**Resilience4j** (per D-054 + D-056 implications and `feature-inventory-target.md` line 387). In the absence of the constraint, Resilience4j is the natural fit: JVM-native, modular (circuit-breaker + bulkhead + rate-limiter + retry are separate decorators), actively maintained (latest release 2026-04 per /web-research 2026-05-23). Wiring path against `stage-4.0` services: the `resilience4j-spring-boot3` starter remains the canonical Spring Boot 4-compatible entry point as of `last_verified` (a SB 4 starter is in-flight upstream — surface to instructor if it has GA'd by cohort delivery). SB 4.0 ships native `@Retryable` + `@ConcurrencyLimit` annotations that cover the simple cases; Resilience4j is the answer when you need full circuit-breaker semantics + Micrometer/OTel metrics. Resilience4j remains the canonical W5 AIOps anchor.

## 3. Candidate alternatives the pair must evaluate

| Alternative | One-line capability | Maturity (per `tech-research` recency policy) |
|-------------|---------------------|-----------------------------------------------|
| **Hystrix (legacy / archived)** | Netflix's circuit-breaker library — the historical standard | **Archived 2018-11** (per Netflix Tech Blog, retrieved 2026-05-23 via /web-research); maintenance-mode only; **known-bad-pattern: do NOT adopt for new work** |
| **Hand-rolled circuit + bulkhead** | Custom Java implementation of circuit-breaker + bulkhead patterns | Stable (you control it), but: who maintains it in 2 years? What's the test coverage story? FedRAMP scrutiny on custom security-adjacent code is real |
| *(optional)* **Spring Cloud Circuit Breaker** | Spring's abstraction layer that wraps Resilience4j (default) or other backends | Stable; useful if pair wants vendor-neutrality, but adds a layer of indirection over what is functionally Resilience4j |

## 4. Comparison dimensions the pair must score

Per `templates/scenario-alternatives.md` §4: capability fit, cost shape, operational burden, security posture, maturity, exit cost. Score 1–5 per dimension per alternative.

**Specific dimensions to weight heavily for this scenario:**

- **Maturity** — Hystrix is archived; this is a 1-point disqualifier unless the pair argues "we already have Hystrix in another Karsun engagement and we want to consolidate." Defending Hystrix here requires real argument.
- **OpenTelemetry integration** — W5 work depends on this. Resilience4j has first-class Micrometer metrics + OTel integration. Hand-rolled means you build the metrics yourself.
- **Spring Boot 4.x compatible** — post-Thu hop, the services are SB 4.0 + Java 21. The `resilience4j-spring-boot3` starter is the load-bearing Spring Boot-4-compatible entry point as of `last_verified`; the pair should name whether a SB 4 native starter has GA'd or whether the SB 3 starter is the route forward.

## 8. Anti-patterns to call out in defence

- **"Hystrix is what the previous prime used"** — without addressing the 2018 archive notice. Pair must explicitly defend OR disqualify.
- **"Hand-rolled because we want control"** — without addressing who maintains it in 2 years and the FedRAMP custom-code review burden.
- **Treating circuit-breaker as a one-shot library decision** — circuit-breaker, bulkhead, rate-limiter, and retry are 4 separate Resilience4j modules; the choice IS modular, and the pair should defend each module independently.
- **Ignoring the W5 hand-off** — the scenario IS the W5 AIOps Resilience4j anchor (per `feature-inventory-target.md` line 387). The ADR must name the W5 ticket it sets up.
