# W02-SA-1 — Vector store choice: Atlas Vector Search vs pgvector vs Pinecone

## 1. The client constraint

> *"Our acquire-gov training-project already has MongoDB Atlas in the stack — it holds the document store today. The Tuesday work added Vector Search on top. But our agency CIO got a vendor pitch that Postgres pgvector would be simpler (we're already on Postgres for relational data) and a separate pitch that Pinecone serverless is the 'production-grade' option. Meanwhile our `requirements.txt` lists `pinecone-client==5.0.0` with no actual import — debt Item 7 — and the `/admin/config` screen lies and says 'pinecone available.' Defend the keep-or-remove call on Atlas Vector Search this week. We're locked in or we're not — show me the comparison."*
>
> — Karsun delivery lead, Mon W2 EOD

## 2. The default programme answer

**MongoDB Atlas Vector Search** (per ADR-0003 in the training-project; locked-in default per D-031 — collocation with the document store reduces FedRAMP data-residency surface). Hybrid retrieval via `$vectorSearch` + `$search` + `$rankFusion` (built-in RRF). Pre-filter on `agency_id` declared as a `filter` field in the index definition closes debt Item 10 at the index level.

## 3. Candidate alternatives the pair researches Wed W2

| Alternative | One-line capability | Maturity (per `tech-research` recency policy) |
|-------------|---------------------|-----------------------------------------------|
| **PostgreSQL pgvector** | Postgres extension for vector columns + ANN search (HNSW + IVF); same DB instance as relational data | Stable + extension v0.7+ widely deployed (foundation-stable 12-month window) |
| **Pinecone serverless** | Managed vector DB; v1 GA 2024; "vector-native" architecture; namespace-per-tenant model | Stable + serverless tier GA per Pinecone docs retrieved 2026-05-23 (hot-tech 3-month window) |

## 4. Comparison dimensions the pair must score

For each alternative (and the Atlas default), score 1–5 on:

- **Capability fit** — Hybrid retrieval out-of-box? Index-level filter for multi-tenant? Atlas RRF is built-in; pgvector requires manual UNION + RRF math; Pinecone has metadata filter but RRF is application-side.
- **Cost shape** — Atlas: bundled with existing M30 cluster (no marginal cost up to a vector-storage cap); pgvector: free (already on Postgres); Pinecone: $0.10/M reads + storage + serverless minimums.
- **Operational burden** — Atlas: one cluster to manage (already there); pgvector: Postgres tuning is well-understood but HNSW index rebuild on large corpora is painful; Pinecone: zero ops but vendor-locked metric scope.
- **Security posture** — FedRAMP boundary: Atlas (yes, in-boundary with MongoDB Atlas for Government); pgvector (yes, your own Postgres); Pinecone (FedRAMP gap as of 2026-Q1 per Pinecone trust center retrieved 2026-05-23 — **THIS IS THE LEAD DEFENSE POINT**); multi-tenant: Atlas pre-filter index, pgvector WHERE-clause, Pinecone namespaces (see `pinecone-namespaces-as-multitenancy` known-bad-pattern — namespaces alone aren't enough at scale).
- **Maturity** — Atlas Vector Search GA 2024-06; pgvector v0.7+ widely deployed; Pinecone serverless GA 2024.
- **Exit cost** — Atlas → pgvector: re-embed + re-index + RRF re-implementation; Atlas → Pinecone: re-index + filter-pattern rewrite + FedRAMP review. None are trivial; pgvector exit is cheapest because the embeddings are portable + Postgres operationally familiar.

## 5. Required pair deliverable (in their pair-project repo)

A short ADR (≤2 pages) at `adrs/W02-SA-1.md`:

- **Decision:** keep Atlas Vector Search OR migrate to pgvector OR migrate to Pinecone — and remove `pinecone-client` dep regardless of decision (debt Item 7 closure).

## 8. Anti-patterns to call out in defence

- **"We pick Atlas because it's already in the stack."** Defaulting on inertia is not a defense. The exercise is *honest comparison*. If the pair concludes pgvector genuinely wins on relational-vector join needs, they should say so — but they need the requirement evidence to back it.
- **"Pinecone is more production-grade."** Pinecone has FedRAMP gap as of Q1 2026. For federal data this is not a marketing-vs-engineering trade — it's a compliance show-stopper. Defender must address this explicitly.
- **"pgvector is free."** Postgres operations cost is not zero. HNSW index rebuild on FAR+DFARS (3,500 pages → ~25K chunks) is multi-hour. Free dollars hide real engineering time.
- **`pinecone-namespaces-as-multitenancy`** (known-bad-pattern). Namespaces alone are not a tenancy boundary at production scale. If a pair argues for Pinecone with namespace-only tenancy, push back.
- **Treating "FedRAMP Authorized" as binary.** FedRAMP Moderate vs High vs Boundary inclusion is the real question for federal-acq workloads. Don't conflate.
