# W02-SA-3 — Embedding model selection: Titan Embeddings v2 vs domain-tuned vs embedding-routing

## 1. The client constraint

> *"We're going to index two different content types in the same RAG layer this programme. **FAR/DFARS clause text** is one — formal regulatory English with consistent structure. **Vendor past-performance narratives from CPARs** are the other — free-form English, sometimes including domain jargon, written by Program Managers. Should they share an embedding model? If they share, we lose some quality on each. If they route to different models, we get better quality but pay infrastructure cost — and the W5 AIOps observability dial gets more complicated. The eval harness Fri lets us test this. Defend your choice."*
>
> — Karsun delivery lead + Program Manager, Mon W2 EOD

## 2. The default programme answer

**AWS Bedrock Titan Embeddings v2** (1024-dim default) for everything — clause text + CPAR narratives + future content types. Per ADR-0004 in the training-project. Single model: one cost line item, one operational surface, one quality dial to tune. Reasoning: Titan v2 is general-purpose strong on English; the cohort hasn't yet seen evidence that a domain-tuned model wins on FAR/DFARS specifically (no public eval data; cohort can produce it Fri).

## 3. Candidate alternatives the pair researches Wed W2

| Alternative | One-line capability | Maturity (per `tech-research` recency policy) |
|-------------|---------------------|-----------------------------------------------|
| **Cohere Embed v3 multilingual on Bedrock** | Domain-flexible embedding with explicit `input_type` parameter (`search_document` vs `search_query` vs `classification`) | Stable on Bedrock per AWS docs retrieved 2026-05-23 |
| **Embedding-routing (Titan v2 for clauses, Cohere Embed v3 for CPARs)** | Different model per content type; routing function picks based on `content_type` metadata at ingestion + query time | Pattern; both underlying models stable; routing is application-side logic |

## 4. Comparison dimensions the pair must score

For each alternative (and the Titan v2 default), score 1–5 on:

- **Capability fit** — Titan v2: general English, no `input_type` param, unit-normalized output (cosine = dot); Cohere v3: explicit `input_type` (asymmetric search), 1024-dim default, returns unit-normalized; routing: best-of-both for the content type.
- **Cost shape** — Titan v2: $0.02 per 1M input tokens (per AWS pricing page retrieved 2026-05-23); Cohere v3: $0.10 per 1M input tokens; routing: weighted average per content-type mix (~70% clauses, 30% CPARs in expected production load).
- **Operational burden** — Single model: one Bedrock invocation point; routing: a router function that needs tests + must not silently break when a new content type is added; embedding-version migration burden (re-embed entire corpus when model upgrades) is per-model.
- **Quality (preliminary, pre-Fri-harness)** — No FAR/DFARS-specific public eval data; Titan v2 reportedly strong on regulatory English (AWS-cited evals); Cohere v3 asymmetric search may help on the query-document distinction; routing in theory wins but adds complexity.
- **FedRAMP boundary** — Titan v2: in-boundary on Bedrock; Cohere v3 on Bedrock: in-boundary (Bedrock-routed); routing: in-boundary if both legs in-boundary.
- **W5 AIOps observability dial** — Single model: one cost dimension to track; routing: per-content-type cost dimensions (more granular, more useful for W5 governance — but more to instrument).

## 5. Required pair deliverable (in their pair-project repo)

A short ADR (≤2 pages) at `adrs/W02-SA-3.md`:

- **Decision:** Titan v2 only OR Cohere v3 only OR routing — with explicit reasoning about clause-text vs CPAR-narrative quality trade-off.
- **Context:** two content types, no public FAR-specific eval data yet, Fri harness will produce it.
- **Considered:** brief notes on each alternative, scored 1–5 across the six dimensions above.
- **Consequences:** what the pair gives up. Single Titan v2: lose any routing-quality lift. Single Cohere v3: pay 5x cost without proven quality lift for clause text. Routing: complexity + per-model-deprecation surface.
- **Rollback story:** if Fri eval shows current pick regresses, what's the swap-out cost? (Re-embedding ~25K chunks is multi-hour but tractable; routing change is a single-file router rewrite.)
- **References:** dated citations.

## 8. Anti-patterns to call out in defence

- **"More expensive must mean better."** Cohere v3 at 5x cost is not 5x better on regulatory English (no public eval data either way). Cost-per-quality is the dimension, not absolute cost.
- **"Routing is overengineering for week 2."** Defensible IF the pair has explicit threshold: "if Fri eval shows >10% quality lift from routing, we adopt." Indefensible if the pair never names a re-evaluation trigger.
- **`vector-cosine-default`** (known-bad-pattern). If a pair argues for cosine as "the standard," push back — Titan v2 unit-normalizes (cosine = dot), Cohere v3 does too; for non-normalized models you'd want dot or L2. Don't pattern-match.
- **"MTEB leaderboard says X."** MTEB is general-purpose. The cohort has no FAR-specific benchmark. Citing MTEB as definitive for the regulatory domain is misuse.
- **Embedding-version-migration story missing.** Bedrock model deprecation is real (per `bedrock-old-model-ids` known-bad-pattern — Claude v2 and Claude Instant IDs were deprecated). Embedding-version migration = re-embed everything. The ADR must include this risk.
