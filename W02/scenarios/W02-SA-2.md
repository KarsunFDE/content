# W02-SA-2 — Hybrid retrieval strategy: BM25+dense vs reranker-only vs full hybrid+rerank

## 1. The client constraint

> *"On `POST /rag/clause-search` we have a p95 latency budget under 800ms for the agency CIO's pilot. The clause text has unusual properties — FAR/DFARS clause IDs are exact lexical matches (vendor types '52.212-4' verbatim), but the natural-language paraphrase queries ('what's the timing on streamlined acquisitions?') need semantic understanding. The Cohere Rerank 3.5 model on Bedrock adds ~150-300ms latency but Anthropic's contextual-retrieval research suggests reranker is the highest-quality lift. Defend your retrieval strategy. The eval harness ships Fri — pick the strategy that wins on quality without blowing the latency budget."*
>
> — Contracting Officer via Karsun delivery lead, Mon W2 EOD

## 2. The default programme answer

**Hybrid (BM25 + dense via Atlas `$rankFusion`) + Cohere Rerank 3.5 via Bedrock as a final pass.** Atlas's built-in `$rankFusion` (RRF, k=60) merges `$vectorSearch` and `$search` rankings; top-20 hybrid results go to Cohere Rerank 3.5; top-5 reranked feed the LLM. Budget breakdown (preliminary, pre-eval): 200ms Atlas retrieval + 250ms reranker = 450ms RAG layer; LLM call adds 300-400ms; total ~750-850ms p95 — at the budget ceiling.

## 4. Comparison dimensions the pair must score

For each alternative (and the hybrid+rerank default), score 1–5 on:

- **Capability fit** — How well does each handle (a) exact clause-ID match (vendor types "52.212-4"), (b) semantic paraphrase ("when's the proposal due?"), (c) dual-corpus precedence (FAR + DFARS supplement detection)?
- **Cost shape** — Atlas retrieval cost is baked into the cluster; Cohere Rerank 3.5 charges per 1K queries on Bedrock (cohort calculates expected QPS × monthly query budget); reranker adds variable cost; multi-query retrieval (if added) multiplies LLM-as-paraphraser cost.
- **Operational burden** — One retrieval path (dense+rerank) vs two retrieval paths (BM25+dense+rerank) vs hybrid (Atlas does the merge — one path from caller's perspective); reranker model deprecation risk on Bedrock catalog (hot-tech 3-month window).
- **Latency posture** — Atlas retrieval ~200ms p95 baseline; reranker adds 150-300ms; BM25 alone ~50ms; dense alone ~150ms; hybrid retrieval ~200ms (Atlas does it as one query).
- **Maturity** — Atlas `$rankFusion` GA; Cohere Rerank 3.5 GA on Bedrock; pure-dense + BM25-only patterns well-established.
- **Eval quality (preliminary, pre-Fri-harness)** — Anthropic's contextual-retrieval ablation table is the strongest public data: hybrid + rerank = 67% failure reduction over baseline dense; hybrid alone = 49%; reranker over dense alone = 56%. *FAR/DFARS-specific eval data does not exist — Fri's harness gives us that.*

## 8. Anti-patterns to call out in defence

- **"Reranker is always worth it."** Reranker latency might exceed the budget for some queries; on exact-clause-ID lookups (where BM25 alone is sufficient), the reranker burns latency for zero quality lift.
- **"BM25 is dead, dense is the future."** BM25 still beats dense on exact-string-match queries — FAR clause IDs are the canonical example. Anyone arguing dense-alone for the FAR domain has not thought about the lexical-specificity of the corpus.
- **"We'll let the eval harness pick for us."** The eval harness comes Fri; ADRs are due Thu EOD. The pair has to pick on principled reasoning + Wed's research, then validate Fri. If the harness vindicates Mon's plan, great. If not, that's a W3 §0 retro learning.
- **`ragas-faithfulness-only`** (known-bad-pattern). If a pair argues "the metric to optimize is faithfulness," push back — recall and precision matter too, especially for dual-corpus retrieval.
- **Treating latency budget as average not p95.** Average latency might be fine; p95 is what blows the SLO.
