# W01-SA-2 — Server-rendered Java UI (Thymeleaf) vs Angular SPA

## Client constraint

> *"We're stepping into a federal civilian agency engagement that just lost their previous Java contractor. Their handoff package is a Spring Boot 2.x monolith with Thymeleaf for the UI — 60 templates, all the contracting-officer workflows on the server side. Their program manager wants to know why your team is pitching an Angular SPA front-end when their previous team delivered everything as Spring + Thymeleaf at half the headcount. She's asking: 'Will Thymeleaf hand off cheaper at the end of our engagement than Angular?' She doesn't actually care about which is technically better — she cares about who's on the hook to maintain it after we're gone."*
>
> — Program Manager (federal civilian agency, FAR Part 17 task-order vehicle), W1 Thu 2026-05-28

## The programme default

**Angular 17+ SPA** (per training-project + the 3 pair-project repos — all 4 brownfields use Angular at the edge). Defended on the basis that contracting-officer workflows are interactive (live form validation, multi-step solicitation drafting, streaming AI-assisted suggestions per W1 Fri SSE work), the cohort's existing Java + Python service mesh is API-shaped, and Angular's CLI + TypeScript discipline matches the rigor the cohort's been practising on the backend.

## Candidate alternatives the pair researches Wed W2

1. **Spring Boot + Thymeleaf (server-rendered)** — Thymeleaf is a Spring-native template engine bundled in the Spring Boot starter ecosystem. Pages render on the server; interactivity comes from sprinkled jQuery / vanilla JS or full-page reloads. **Defendable for federal-handoff scenarios** where the next contractor is also a Java shop. **Less defendable when the UX is conversational AI-assisted drafting** — the cohort's pair-projects all need streaming LLM output, which is harder server-rendered.
2. **HTMX + Spring Boot (progressive enhancement)** — Modern middle ground. Server still renders HTML; HTMX adds partial-page swaps + SSE-driven updates without a JS framework. Smaller blast-radius than Angular, more interactive than pure Thymeleaf. **Federal-handoff appeal**: still all-Java skill set for the next team. **Risk**: HTMX maturity + library ecosystem is narrower than Angular; less standard at federal contractors.

## Pair's deliverable (ADR + Live Defense)

The pair authors an ADR by Thu W2 EOD covering:

- **Decision** — does the pair stand by Angular for this engagement, or recommend Thymeleaf / HTMX for the federal-handoff posture?
- **Context** — who's on the hook to maintain this at hand-off? What's the interactivity profile of the workflows (form-heavy CRUD vs AI-assisted drafting vs real-time dashboard)? What's the contractor pool the agency would hire next?
- **Alternatives** — both alternatives evaluated honestly. The pair must engage with the PM's actual question (handoff cost), not deflect to technical sophistication.
- **Consequences** — what happens at hand-off if the pair recommends Angular and the agency hires a pure-Java shop next? What happens if the pair recommends Thymeleaf and the AI-streaming UX never lands?
- **Rollback story** — can the pair scaffold the UI in Thymeleaf for the form-heavy workflows AND Angular for the AI-assisted drafting flows? Where's the seam?

## Anti-patterns to call out in defence

- **Treating "modern stack" as the answer to the handoff question.** The PM asked about maintenance cost; "Angular is more modern" doesn't answer her question. Pair must reason from her constraint, not their preference.
- **Dismissing Thymeleaf as legacy.** Thymeleaf 3.x ships in current Spring Boot starters; it isn't deprecated. Calling it "legacy" reveals the pair didn't actually research it. Defender loses credibility.
- **Ignoring the AI-streaming-UX requirement.** This isn't a 2015 CRUD app — pair-projects all do streaming LLM output (W1 Fri SSE work). Pair must defend whichever UI they pick *against the streaming requirement*, not handwave past it.
- **Splitting the difference without thinking through the seam.** "We'll use both" is a valid answer ONLY if the pair can name the seam, the routing rule, and the handoff implication of the boundary. Hand-waving "we'll use Thymeleaf for some pages and Angular for others" without defending the line is weaker than picking one.
- **Conflating Angular complexity with team skill.** "The Karsun-side team is comfortable with Angular" is not a defense to a federal customer at hand-off — *their* next contractor isn't on Karsun's bench.

## Probe questions for Live Defense

The defender should be ready for:

1. *"The PM is hiring her next contractor from a pool that's 90% Java shops. Why should she pay a JavaScript-skilled contractor's premium to maintain your Angular SPA after you're gone?"*
2. *"Show me a Thymeleaf page that does streaming LLM-assisted draft suggestions as well as your Angular component does. If you can't, what's your actual answer?"*
3. *"Spring Boot 3.x ships `spring-boot-starter-thymeleaf` — that's not abandoned. Why did you assume Thymeleaf is legacy?"*
4. *"You proposed HTMX. Show me a federal customer reference where it's been delivered in a FedRAMP-boundary system. If you can't, are you sure it's a safe recommendation to make?"*
5. *"If we split — Thymeleaf for form-heavy CRUD pages, Angular for the AI-drafting flow — what's the auth boundary between them? Same JWT? Different sessions? Walk me through it."*
6. *"The PM asked about handoff cost. What number do you give her?"* (The defender doesn't have to have a real number — they have to have *a framework for thinking about it* + the courage to defend a directional answer.)
