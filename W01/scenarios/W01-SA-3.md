# W01-SA-3 — Synchronous vs streaming SSE for solicitation drafting UX

## Client constraint

> *"Two stakeholders are pulling our team in opposite directions. The agency CIO sat through a ChatGPT demo last week and now insists the LLM-assisted solicitation drafting feature 'must stream like ChatGPT does — the contracting officers shouldn't sit and wait.' Meanwhile the agency's integration team — the folks who own the downstream procurement system that pulls completed drafts via our API — has a hard requirement that 'every draft is delivered atomically; partial drafts are not consumable by the downstream system.' Their integration contracts assume request/response. We have one week before the pilot demo to architect the endpoint. What do we ship?"*
>
> — Karsun engagement lead relaying agency CIO + agency integration team, W1 Thu 2026-05-28

## The programme default

**SSE (Server-Sent Events) streaming** for the contracting-officer UI surface (per W1 Fri practical — the cohort wires `services/ai-orchestrator/` async FastAPI streaming through the API gateway to the Angular component). SSE is HTTP-native, one-direction (server → client), reconnects automatically, and works over standard HTTP/2 + HTTP/3 multiplexing without WebSocket upgrade negotiation. The Spring Boot side uses `SseEmitter` to bridge between the Java service mesh and the Python orchestrator.

## Candidate alternatives the pair researches Wed W2

1. **Synchronous JSON request/response (atomic)** — Single HTTP POST → full draft JSON in response body. Trivially compatible with the agency integration team's contract (atomic delivery). UX is "click → loading spinner for 20-40 seconds → draft appears". **Defendable when**: the user pool tolerates the wait, the integration contract is hard, the operations team needs simple retry semantics. **Risk**: the CIO will reject it on UX grounds; the demo will feel slow next to ChatGPT.
2. **WebSocket bidirectional** — Persistent connection, server-pushes-tokens-as-they-arrive AND client-pushes-mid-stream-corrections. Strictly more capable than SSE. **Defendable when**: the UX is genuinely interactive mid-generation (user steers the draft, accepts suggestions, rejects sections). **Risk**: deployment + load-balancer + WAF complexity is higher (sticky sessions, connection state, FedRAMP boundary penetration testing), the integration team's contract is even further away from atomic delivery, the W1-Fri-shipped Spring Boot stack uses `SseEmitter` not `@MessageMapping`.

## Pair's deliverable (ADR + Live Defense)

The pair authors an ADR by Thu W2 EOD covering:

- **Decision** — does the pair recommend SSE streaming, synchronous atomic, WebSocket bidirectional, or a hybrid (e.g., stream to the UI surface but emit a single atomic event to the integration system at completion)?
- **Context** — name *both* stakeholders explicitly. UX experience for the contracting officer. Integration contract with the downstream procurement system. FedRAMP boundary implications.
- **Alternatives** — all three candidates evaluated. Pair must name what each costs: deployment complexity, operations runbook, learner-debugging difficulty, FedRAMP penetration-test surface.
- **Consequences** — if the pair picks SSE-for-UI + atomic-for-integration as a hybrid, what's the contract between them? When does the integration event fire? What happens if the SSE stream errors mid-generation — does the integration system get a partial event or no event?
- **Rollback story** — if SSE turns out to be unworkable for some agency network constraint (corporate proxy that buffers streaming responses, etc.), can the pair fall back to synchronous without UI rewrite?

## Anti-patterns to call out in defence

- **Picking SSE because the training stack already has it.** That's tech-loyalty, not architectural reasoning. The defender must name *why SSE wins for this client's specific shape*.
- **Treating the integration team's "atomic delivery" requirement as a stakeholder problem to negotiate away.** It's likely a contract requirement of the downstream procurement system — federal-acquisitions integration is often FAR-Part-39-shaped + has real audit-trail implications. Defender who says "just convince them" without engaging with *why* they need atomic delivery loses credibility.
- **Proposing WebSocket without naming the operational cost.** WebSocket adds sticky-session load-balancer config, connection state, longer-running connections through FedRAMP-boundary firewalls. If the pair recommends WebSocket they must defend each of those.
- **Conflating "ChatGPT-like UX" with "must be streaming."** ChatGPT's UX is streaming because the model takes time. If the drafting model returns in 3-5 seconds, a fast spinner is arguably *better UX* than partial streamed output that learners have to wait to finish anyway. Pair must address the actual latency envelope of the model.
- **Ignoring the hybrid option.** SSE-to-UI + atomic-event-to-integration-system is a real architectural answer the pair should consider. Defaulting to "pick one" without thinking through a split misses the most likely production answer.
- **Skipping retry semantics.** Whatever the pair picks, they should be able to answer "what happens if the connection drops at token 47 of 200?"

## Probe questions for Live Defense

The defender should be ready for:

1. *"You picked SSE. Show me your operations runbook for 'the contracting officer's session died at token 47.' Where does the partial output go? Does the integration system get a partial event?"*
2. *"The integration team's contract says 'atomic delivery.' What specifically would you commit to them — a guaranteed callback on completion? An idempotency key? A finalize endpoint they call after the stream ends? Walk me through the API surface."*
3. *"If our average draft is 4 seconds end-to-end, is streaming actually better UX than a 4-second spinner? Defend your answer with real-user behavior, not vibes."*
4. *"You proposed a hybrid (SSE-to-UI, atomic-to-integration). When does the integration event fire — at first token, at last token, at first error, or only on full completion? What happens to the partial work if the user cancels mid-stream?"*
5. *"FedRAMP penetration testing has a specific protocol for long-lived connections. How does SSE affect your pen-test surface vs synchronous POST? Vs WebSocket?"*
6. *"The agency's corporate proxy buffers HTTP response bodies up to 1MB before flushing. What does that do to your SSE stream? Have you tested it?"*
7. *"Reasonable engineer reads your ADR cold. What's the one thing they need to take away — not the decision itself, but the reasoning shape they could apply to the next streaming-vs-atomic call?"*
