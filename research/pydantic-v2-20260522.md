# Tech research brief — Pydantic v2

Last verified: 2026-05-22 · Recency window applied: foundation-stable 12mo (with last-3-mo flag for v2.11 ramping items) · Pinned version: v2.11.x

## 1. What it is (3–5 sentences, no jargon)

Pydantic is the de facto Python data-validation library: declare typed models, get parsing + validation + serialisation for free. The cohort touches it everywhere — FastAPI request/response schemas in W1, LangChain `create_agent` tool I/O schemas in W1–W2, and RAG document schemas in W2. v2 (Rust-core) is the long-stable line; the cohort writes against v2.11.x but the patterns that matter (field_validator, ValidationInfo.context, model_config = ConfigDict) are stable across the 12-month window.

## 2. Current stable state (as of `last_verified`)

- Latest in v2 line: **v2.11** (focused on build-time performance + alias-config API: new `serialize_by_alias` and `validate_by_alias` ConfigDict settings). v2.10 (2024–2025) loosened the default `protected_namespaces` to `('model_dump', 'model_validate',)` — meaning fields named `model_id` / `model_input` etc. no longer collide with BaseModel internals.
- Active major version line: v2.x. v1 is EOL.
- Breaking changes in last 3 months: **none breaking in v2.11**; `serialize_by_alias=False` keeps v2.10 behaviour for back-compat. Flag-as-ramping: new alias-config knobs.
- Known-bad-pattern check: **pass**.

### Canonical v2 patterns the cohort uses

- **`field_validator` (replaces v1 `@validator`):** decorator on a classmethod, takes the field name(s) and a `mode` (`before` | `after` | `plain` | `wrap`). Pass `info: ValidationInfo` as the second arg if you need context.
- **`ValidationInfo.context`:** opaque dict passed at validation time via `Model.model_validate(data, context={...})`. Field validators read it via `info.context`. **Limitation:** context cannot be passed via the direct `Model(...)` constructor — only via `model_validate()`.
- **`model_config = ConfigDict(...)`:** the v2 idiom for model-level configuration. Replaces v1's inner `class Config:`. Common knobs: `frozen=True`, `from_attributes=True` (ORM mode), `populate_by_name=True`, `str_strip_whitespace=True`, `extra="forbid"`, `validate_by_alias=True`, `serialize_by_alias=True`.

## 3. What we teach (and what we deliberately don't)

- **In scope (W1–W2):** `BaseModel` with type-annotated fields; `field_validator(..., mode='after')` for normalising/checking values; `ValidationInfo.context` for passing tenant/user context into validation (e.g., role-aware schema constraints in the FastAPI dependency layer); `model_config = ConfigDict(extra='forbid', frozen=True)` for strict tool-input schemas in LangChain `create_agent`; `model_validate(data, context=...)` as the validation entry point when context is needed.
- **Out of scope:** v1 compat layer (`pydantic.v1`); custom root validators (use `model_validator` instead); Pydantic Settings beyond the one-slide FastAPI config mention.
- **Misconceptions to pre-empt:** (a) "`@validator` still works." — deprecated; use `field_validator`. (b) "I can pass context through `Model(...)`." — false; only `model_validate(data, context=...)`. (c) "`class Config:` is the v2 way." — false; use `model_config = ConfigDict(...)`.

## 5. Code reference snippets (idiomatic, current API)

```python
from pydantic import BaseModel, ConfigDict, ValidationInfo, field_validator

class PolicyQuery(BaseModel):
    model_config = ConfigDict(extra="forbid", frozen=True)

    tenant_id: str
    query: str

    @field_validator("tenant_id", mode="after")
    @classmethod
    def check_tenant_access(cls, v: str, info: ValidationInfo) -> str:
        ctx = info.context or {}
        allowed = ctx.get("allowed_tenants", set())
        if allowed and v not in allowed:
            raise ValueError(f"tenant_id {v} not in caller's allow-list")
        return v

# Caller wires context in at validate-time, NOT at construction
PolicyQuery.model_validate(
    {"tenant_id": "tenant-abc", "query": "..."},
    context={"allowed_tenants": {"tenant-abc", "tenant-xyz"}},
)
```

## 6. Risks and watch-items

- The "context only via model_validate" gotcha trips people. Worth a single slide.
- LangChain `create_agent` tool-input schemas use Pydantic — pin v2.x in cohort requirements (≥2.10 for the protected_namespaces default we want).
- v3.0 is on the long horizon; not relevant to W1–W2 but worth watching for cohort v2.

## 7. Alternatives the cohort should be aware of

- `dataclasses` (stdlib) + manual validation — surfaced as the "why Pydantic" baseline.
- `attrs` — mentioned only as a name-recognition footnote.
